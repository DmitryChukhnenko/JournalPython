## б) Постановка задачи

Разработать консольное приложение на языке Python для автоматизации ведения учебного расписания высшего учебного заведения. Система должна обеспечивать полный цикл работы с данными: загрузку, создание, редактирование, удаление и долгосрочное хранение информации о ключевых сущностях учебного процесса.

### Функциональные требования
1. **Управление справочниками:**
   - Учебные группы (название, количество студентов, куратор)
   - Предметы/дисциплины (название, код предмета, часы)
   - Аудитории (номер, корпус, вместимость)
   - Преподаватели (ФИО, электронная почта, кафедра)
2. **Формирование расписания:**
   - Привязка занятия к конкретной группе, предмету, преподавателю и аудитории
   - Указание дня недели и временного интервала
   - Автоматическая проверка на конфликты (занятость преподавателя, группы, аудитории в одно время)
3. **Хранение данных:**
   - Автоматическое сохранение в файл формата JSON
   - Загрузка существующих данных при запуске приложения
   - Корректная обработка отсутствия файла или повреждённых данных
4. **Пользовательский интерфейс:**
   - Текстовое консольное меню с чёткой навигацией
   - Валидация пользовательского ввода на каждом этапе
   - Таблицы и структурированный вывод для удобства восприятия

### Нефункциональные и технические требования
- **Язык и стек:** Python 3.8+, использование исключительно стандартной библиотеки (`json`, `uuid`, `os`, `pathlib`, `typing`, `datetime` и др.)
- **Парадигма:** Объектно-ориентированное программирование с обязательной реализацией инкапсуляции. Наследование и полиморфизм применяются обоснованно, без искусственного усложнения.
- **Архитектура:** Разделение ответственности по слоям (Model–Repository–Core–UI). Вынос логики валидации и преобразования данных в чистые функции (принципы функционального программирования).
- **Документирование:** Полное покрытие кода `type hints`, docstrings (Google style), поясняющие комментарии к нетривиальным участкам.

## в) Анализ задачи и структура классов

### 1. Предметный анализ
Домен системы описывается четырьмя базовыми сущностями: `Группа`, `Предмет`, `Аудитория`, `Преподаватель`. Каждая обладает уникальным идентификатором и набором атрибутов с жёсткими типами. Занятие (`ScheduleSlot`) выступает составной сущностью, агрегирующей идентификаторы базовых сущностей и временные параметры. Бизнес-логика сосредоточена в менеджере расписания, который координирует операции CRUD и обеспечивает целостность данных через проверку конфликтов.

### 2. Применение принципов ООП
| Принцип | Реализация в проекте |
|---------|----------------------|
| **Инкапсуляция** | Все атрибуты моделей и менеджеров объявлены приватными (префикс `_`). Доступ осуществляется только через `@property` и сеттеры, внутри которых выполняется валидация типов, диапазонов и форматов. Внешний код не может нарушить инварианты объектов. |
| **Наследование** | Базовый класс `BaseEntity` выносит общую логику: генерацию `uuid`, методы сериализации `to_dict()`, `from_dict()`, метод `__str__()` и шаблонный метод `_validate()`. Все доменные модели (`Teacher`, `Group`, `Subject`, `Classroom`) наследуют от него, устраняя дублирование кода. |
| **Полиморфизм** | Унифицированный интерфейс работы с коллекциями: все сущности реализуют одинаковые методы `to_dict()`, `from_dict()`, `__str__()`. Репозиторий и менеджер работают с `list[BaseEntity]`, не завися от конкретного типа дочернего класса. Методы формата вывода и парсинга адаптируются под тип сущности динамически. |

### 3. Функциональное разделение
Для сохранения чистоты ООП-моделей и избежания побочных эффектов валидация и преобразование данных вынесены в отдельные модули:
- `core/validators.py` содержит только чистые функции (без `self`, без изменения внешнего состояния): проверка формата времени, выявление пересечений интервалов, генерация отчётов о конфликтах.
- `ui/` содержит утилиты ввода/вывода, преобразующие сырые строки в структурированные словари, которые затем передаются в конструкторы моделей.

### 4. Иерархия и структура классов
```text
BaseEntity (models/base_entity.py)
    ├── Teacher (models/teacher.py)
    ├── Group (models/group.py)
    ├── Subject (models/subject.py)
    └── Classroom (models/classroom.py)

ScheduleSlot (core/schedule_slot.py)      # Агрегирует ID сущностей + время
ScheduleManager (core/schedule_manager.py) # Бизнес-логика, CRUD, проверка конфликтов

ScheduleRepository (repository/json_repo.py) # Абстракция хранения (JSON)
ConsoleApp (ui/console_app.py)               # Презентационный слой
```

#### Краткое описание классов
| Класс / Модуль | Назначение | Ключевые методы |
|----------------|------------|-----------------|
| `BaseEntity` | Базовый контракт для всех сущностей | `id`, `to_dict()`, `from_dict()`, `_validate()`, `__str__()` |
| `Teacher`, `Group`, `Subject`, `Classroom` | Доменные модели с валидацией полей | Сеттеры с проверками, переопределение `to_dict/from_dict`, специфичные свойства |
| `ScheduleSlot` | Модель одного занятия | Валидация дня недели и формата `HH:MM`, методы сериализации |
| `ScheduleManager` | Оркестратор бизнес-логики | `add_slot()`, `remove_slot()`, `get_by_teacher()`, `get_by_classroom()`, `get_all()`, проверка конфликтов через `validators` |
| `ScheduleRepository` | Слой персистентности | `load_data()`, `save_data()`, обёртки над конкретными сущностями, кэширование в памяти |
| `ConsoleApp` | Консольный интерфейс | Цикл меню, маршрутизация действий, форматированный вывод, обработка ошибок ввода |

## 🚀 Запуск и использование

### Требования
- Python 3.8+
- Только стандартная библиотека (stdlib)

### Установка и запуск
1. Скопируйте файлы проекта в рабочую директорию
2. Запустите приложение:
   ```bash
   python main.py
   ```

### Пример рабочего процесса
После запуска отобразится главное меню:
```
============================================================
              Система управления расписанием
============================================================
[1] Группы       [2] Предметы      [3] Аудитории
[4] Преподаватели [5] Расписание    [0] Выход
Выберите действие:
```

1. **Добавление справочников:**  
   Выберите соответствующий пункт (`[1]`–`[4]`), укажите действие `Добавить`, введите данные. Система автоматически сгенерирует UUID и проверит корректность формата.
2. **Создание занятия:**  
   Перейдите в `[5] Расписание` → `[1] Добавить занятие`. Система предложит выбрать существующие сущности из списка и указать день недели/время. При конфликте расписания будет выведено подробное сообщение с указанием пересекающихся слотов.
3. **Просмотр и фильтрация:**  
   В разделе расписания доступны фильтры по группе, преподавателю или аудитории. Данные выводятся в виде выровненной таблицы.
4. **Выход:**  
   Опция `[0]` автоматически сохраняет всё состояние в `data/schedule.json` и завершает работу.

## 💾 Хранение данных и обработка ошибок

### Формат данных
Все данные сохраняются в `data/schedule.json` в следующей структуре:
```json
{
  "teachers": [...],
  "groups": [...],
  "subjects": [...],
  "classrooms": [...],
  "schedule": [...]
}
```
Файл создаётся автоматически при первом запуске. При повреждении JSON или отсутствии файла система инициализируется с пустым состоянием, не прерывая работу.

---

### Обработка ошибок
Система гарантирует устойчивость к следующим сценариям:
- ❌ Некорректный тип ввода (например, текст вместо числа) → повторный запрос с пояснением
- ⚠️ Конфликт расписания (пересечение времени) → блокировка добавления, вывод деталей конфликта
- 📁 Отсутствие файла данных / битый JSON → graceful fallback к пустому хранилищу
- 🔗 Попытка привязать несуществующую сущность → валидация наличия ID перед созданием слота
- ⌨️ `KeyboardInterrupt` (`Ctrl+C`) → корректное сохранение и завершение

## 🔑 Ключевые архитектурные принципы

1. **Разделение ответственности:** UI не содержит бизнес-логики, репозиторий не знает о правилах валидации расписания, модели отвечают только за состояние и самовалидацию.
2. **Инкапсуляция:** Прямой доступ к внутренним полям запрещён. Все изменения проходят через свойства с встроенными проверками.
3. **Чистые функции:** Валидация времени, поиск конфликтов, парсинг ввода вынесены в отдельные модули. Функции детерминированы и не имеют побочных эффектов.
4. **Расширяемость:** Добавление новой сущности требует наследования от `BaseEntity` и регистрации в репозитории/менеджере без изменения ядра системы.
5. **Стандартизация:** Типизация (`typing`), документация (`docstrings`), единый стиль сериализации обеспечивают поддержку кода и возможность миграции на внешние API в будущем.
